## 1. What Is This Project?

**TravelTour** is a full-stack Vietnamese tourism booking platform. It lets customers browse curated tours (real Vietnam destinations), book them online, pay via **PayOS** (a Vietnamese payment gateway), and get AI-powered travel advice through a built-in chatbot named **Aria** powered by Google's Gemini. There's also a full admin dashboard for managing tours, bookings, users, and reviews.

The entire UI is in **Vietnamese** (vi-VN).

## 2. Architecture: Strict 4-Tier Layered Design

```
┌────────────────────────────────────────────────┐
│  PRESENTATION  │  app/**/page.tsx (RSC)        │
│                │  components/**/*.tsx          │
├────────────────────────────────────────────────┤
│  APPLICATION   │  app/**/actions.ts            │
│                │  ("use server" Server Actions) │
├────────────────────────────────────────────────┤
│  DATA ACCESS   │  data/queries/*.ts (reads)    │
│  (DAL)         │  data/mutations/*.ts (writes) │
│                │  data/dto/*.ts (Zod schemas)  │
├────────────────────────────────────────────────┤
│  INFRASTRUCTURE│  lib/db.ts, lib/auth.ts        │
│                │  lib/supabase/, lib/payos.ts   │
│                │  prisma/schema.prisma          │
└────────────────────────────────────────────────┘
```

**Key rule:** Each layer only calls the layer directly below it. The DAL has zero knowledge of HTTP, Next.js, or React — it's pure TypeScript + Prisma.

## 3. Database Schema (PostgreSQL via Supabase)

### 8 Models:

| Model | Purpose | Key Fields |
|---|---|---|
| `users` | Auth users (merged with Supabase Auth) | `email` (unique), `full_name`, `role`, `password_hash` |
| `destinations` | Travel destinations | `name`, `country`, `description`, `image_url` |
| `tours` | Tour packages | `title`, `price`, `duration`, `max_capacity`, `destination_id` |
| `tour_images` | Multiple images per tour | `tour_id`, `image_url` |
| `bookings` | Customer bookings | `user_id`, `tour_id`, `number_of_people`, `total_price`, `status` |
| `payments` | Payment records (1:1 with bookings) | `booking_id` (unique), `amount`, `payment_status`, `payos_order_code` |
| `reviews` | Tour reviews | `user_id`, `tour_id`, `rating`, `comment` |
| `wishlist` | User wishlist (composite PK) | `(user_id, tour_id)` |
| `promotions` | Promo codes | `code` (unique), `discount_percentage`, `expiration_date` |
| `chat_sessions` | AI chat session per user | `user_id`, `created_at`, `updated_at` |
| `chat_messages` | Individual chat messages | `session_id`, `role` ("user"/"assistant"), `content` |

All IDs are UUIDs (generated by `uuid_generate_v4()`). Supabase Row Level Security (RLS) is enabled on all tables.

## 4. Authentication

- **Supabase Auth** owns sessions (JWT in cookies).
- **Prisma `users` table** owns profile data (`role`, `full_name`).
- `getSession()` merges both sources — reads JWT from Supabase, then fetches role/name from DB.
- `requireAuth()` calls `getSession()` and throws `AuthError` if not logged in.
- **Middleware** (`middleware.ts`) refreshes the Supabase session cookie on every request.
- **Admin gate:** `app/admin/layout.tsx` checks `role === "admin"`, redirects to `/` otherwise. Uses `force-dynamic` to prevent caching.

Two types of Supabase clients:
- **Anon client** (`createServerClient`) — respects RLS
- **Service-role client** (`createServiceClient`) — bypasses RLS (server-only)

## 5. Key Flows

### 5.1 Read Flow (e.g., Tour Detail Page)
```
Browser → Middleware (refresh JWT) → RSC page.tsx
  → data/queries/tours.ts → Prisma → PostgreSQL
  → Returns plain TS object → Streamed as HTML
```

### 5.2 Checkout & Payment Flow
```
Client (CheckoutForm) → Server Action (actions.ts)
  → DAL mutation (payments.ts): requireAuth() → validate DTO → $transaction
    → Create booking (status=pending)
    → Create payment record
    → Call PayOS API → get checkoutUrl
  → Client redirects to PayOS checkout page
```

### 5.3 Payment Webhook (PayOS → confirm)
```
PayOS → POST /api/payos/webhook
  → Verify signature
  → confirmPayosPayment()
    → $transaction: payments.status=completed, bookings.status=confirmed
```

### 5.4 AI Chat (Aria)
```
Client (ChatWidget, useChat hook) → POST /api/chat
  → Build system prompt with real tour/destination data (lib/chat-context.ts)
  → Stream response from Google Gemini AI
  → Save messages to chat_sessions/chat_messages in DB
```

## 8. Coding Conventions

1. **Pages are RSC by default.** Only add `"use client"` when interactivity is needed.
2. **Server Actions are thin.** They call the DAL, catch errors, map to `ActionResult<T>`, and call `revalidatePath()` when needed.
3. **DAL functions never import from `next/`** — they are framework-agnostic.
4. **All mutation inputs are validated** against Zod schemas in `data/dto/`.
5. **All user-scoped functions call `requireAuth()`** at the top.
6. **Errors are typed** — `NotFoundError`, `ValidationError`, `ForbiddenError` with `.code` property.
7. **Prisma client is imported from `lib/db.ts` only** (singleton pattern).
8. **Components are co-located by domain** — `components/admin/`, `components/landing/`, not `components/tables/`.
9. **Query functions use `unstable_cache`** with revalidation tags for Next.js cache.
10. **UI language is Vietnamese** throughout (labels, error messages, chat responses).
